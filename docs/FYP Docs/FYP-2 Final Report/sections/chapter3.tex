\chapter{System Overview}
\label{ch:system-overview}

By the conclusion of Iteration 4, QCanvas has evolved into a multi-layer platform in which the QCanvas Web IDE, the QCanvas conversion SDK, and the QSim simulation SDK each operate as distinct, independently deployable components connected by a FastAPI API layer. This chapter describes the final architecture, design models, and data design of the completed system.

\section{Architectural Design}

QCanvas employs a layered hybrid architecture in which the Next.js frontend handles the user interface and lightweight operations, the FastAPI backend manages compilation and validation, and the QSim SDK executes circuits on simulation backends. Both the \texttt{qcanvas-sdk} and \texttt{qsim-sdk} are packaged as installable Python tools, making their capabilities accessible beyond the Web IDE.

\subsection{Layered Responsibilities}
\begin{itemize}
    \item \textbf{Presentation Layer (Next.js)}: Monaco editor UI, D3.js circuit visualisation, Chart.js results dashboards, gamification achievement panels, community recipe library, and client-side state management via Zustand.
    \item \textbf{API Layer (FastAPI)}: REST endpoints for conversion, validation, and simulation; WebSocket progress streams; structured error handling; JWT authentication; and rate limiting.
    \item \textbf{Conversion Layer (QCanvas SDK)}: AST parsing, intelligent gate mapping, OpenQASM 3.0 generation via \texttt{QASM3Builder}, validation, conversion analytics, and structured logging.
    \item \textbf{Execution Layer (QSim SDK)}: Simulation backends (statevector, density matrix, stabilizer), job scheduling, result aggregation, caching, and performance metrics. Independently installable via \texttt{pip install qsim-sdk} \cite{qsim_sdk_2026}.
    \item \textbf{Persistence Layer}: PostgreSQL for user accounts, projects, files, conversion history, and achievements; Redis for caching and session management.
    \item \textbf{Observability Layer}: Health checks, Prometheus metrics, execution time histograms, and structured logging via Python's \texttt{logging} framework.
\end{itemize}

\subsection{Technology Stack}

\subsubsection{Frontend Technologies}
\begin{itemize}
    \item Next.js 14 with App Router for server-side rendering and routing.
    \item React 18 with TypeScript for type-safe UI component development.
    \item Tailwind CSS for styling with a custom quantum-themed design system.
    \item Monaco Editor for in-browser code editing with custom syntax highlighting and hover providers.
    \item D3.js for 2D circuit visualisation; \texttt{@react-three/fiber} for 3D rendering.
    \item Zustand for lightweight client-side state management.
    \item Chart.js for measurement histogram visualisation.
\end{itemize}

\subsubsection{Backend Technologies}
\begin{itemize}
    \item Python 3.11+ for backend logic and quantum library integration.
    \item FastAPI for the high-performance async API framework.
    \item Uvicorn ASGI server for production deployment.
    \item Pydantic for data validation and serialisation.
    \item SQLAlchemy ORM with PostgreSQL.
    \item Redis for caching and session management.
    \item Alembic for schema migrations.
\end{itemize}

\subsubsection{Quantum Computing Stack}
\begin{itemize}
    \item Cirq 1.0+ for Google quantum framework support \cite{cirq_developers_2023}.
    \item Qiskit 0.44+ for IBM quantum framework support \cite{qiskit}.
    \item PennyLane 0.33+ for Xanadu quantum framework support \cite{bergholm2018pennylane}.
    \item Custom OpenQASM 3.0 generator (\texttt{QASM3Builder}) \cite{cross2021openqasm}.
    \item QSim SDK: Custom statevector, density matrix, and stabilizer backends \cite{qsim_sdk_2026, nielsen2010quantum}.
\end{itemize}

\subsection{Hybrid CPU--QPU Execution Model}

\subsubsection{Orchestration Boundary}
\begin{itemize}
    \item \textbf{CPU Responsibilities}: Circuit construction, parameter preparation, AST parsing, OpenQASM generation, validation, pre/post-processing, and result aggregation.
    \item \textbf{QSim Responsibilities}: Gate-level circuit execution, statevector/density matrix evolution, measurement sampling, and state snapshot generation.
\end{itemize}

\subsubsection{Data Flow}
\begin{enumerate}
    \item User writes circuit in source framework (Qiskit/Cirq/PennyLane) in the Monaco editor.
    \item Frontend sends code to FastAPI backend via REST API.
    \item Backend parses code using AST or runtime execution (detection by import inspection).
    \item QCanvas SDK generates OpenQASM 3.0 intermediate representation via \texttt{QASM3Builder}.
    \item Validation layer checks conformance to the supported gate subset and control flow.
    \item QSim SDK executes circuit on selected backend (statevector/density/stabilizer).
    \item Results are aggregated and returned to frontend via REST response and WebSocket push.
    \item Frontend visualises results: histograms, circuit diagrams, execution statistics.
\end{enumerate}

\subsection{API Data Models}

All API endpoints use Pydantic models for validation and serialisation:

\subsubsection{Request Models}
\begin{itemize}
    \item \texttt{ConversionRequest}: source\_framework, target\_framework, source\_code, style, validate
    \item \texttt{SimulationRequest}: qasm\_code, backend, shots, seed, noise\_model
    \item \texttt{FileRequest}: name, content, language, project\_id
    \item \texttt{RecipeRequest}: title, code, framework, difficulty, tags
\end{itemize}

\subsubsection{Response Models}
\begin{itemize}
    \item \texttt{ConversionResponse}: success, qasm\_code, stats, warnings, errors, timestamp
    \item \texttt{SimulationResponse}: success, results, execution\_time, backend\_info, timestamp
    \item \texttt{ErrorResponse}: error\_code, message, details, suggestions
    \item \texttt{AchievementEvent}: achievement\_id, name, xp\_awarded, new\_level
\end{itemize}
