\section{Algorithm Design}

The core algorithmic pillar of QCanvas is its multi-stage quantum circuit compilation pipeline. The system accepts Python source code in any supported framework and produces validated, executable OpenQASM 3.0 output through the following sequential algorithm:

\begin{enumerate}
    \item \textbf{Detection Phase:} Identify the source framework by inspecting import statements and class usage patterns using Python's \texttt{ast} module. Fall back to runtime execution if AST detection is ambiguous.
    \item \textbf{Parsing Phase:} Traverse the source AST, extracting circuit operations, qubit registers, gate parameters, and measurement instructions.
    \item \textbf{Gate Mapping Phase:} Map detected gate operations to the canonical OpenQASM 3.0 gate set using a lookup table that handles parameter substitution, qubit broadcasting, and controlled-gate modifiers (\texttt{ctrl}, \texttt{inv}, \texttt{pow}).
    \item \textbf{QASM Generation Phase:} The \texttt{QASM3Builder} assembles the final QASM string, emitting header declarations, qubit/register definitions, gate instructions, and measurement blocks.
    \item \textbf{Validation Phase:} The validator performs semantic analysis on the generated QASM, checking against the supported gate subset, identifying unsupported constructs, and emitting line-precise diagnostics.
    \item \textbf{Dispatch Phase:} Validated QASM is dispatched to QSim, which selects the appropriate backend (statevector, density matrix, or stabilizer) and executes the circuit for the requested shot count.
    \item \textbf{Result Aggregation Phase:} Measurement counts, execution time, backend metadata, and optional state snapshots are assembled into a structured response and returned to the frontend.
\end{enumerate}

\section{Sprint-by-Sprint Implementation Progress}

\subsection{Sprint 1: Core Infrastructure and Basic Features}

Sprint 1 established the foundational architecture and implemented essential quantum computing features.

\subsubsection{Core Infrastructure}
\begin{itemize}
    \item \textbf{Backend:} FastAPI application with Pydantic for data validation and SQLAlchemy for ORM.
    \item \textbf{Frontend:} Next.js 14 application with TypeScript and Tailwind CSS.
    \item \textbf{DevOps:} Docker and Docker Compose configuration for consistent development and production environments.
\end{itemize}

\subsubsection{Quantum Features}
\begin{itemize}
    \item \textbf{Framework Support:} Initial support for parsing and converting Cirq, Qiskit, and PennyLane circuits \cite{qiskit, cirq_developers_2023, bergholm2018pennylane}.
    \item \textbf{Standard Gates:} Implementation of all standard quantum gates and modifiers (controls, inverses) \cite{nielsen2010quantum}.
    \item \textbf{Type System:} Complete type system support within the OpenQASM 3.0 intermediate representation \cite{cross2021openqasm}.
    \item \textbf{Control Flow:} Implementation of classical control flow constructs (if-else).
\end{itemize}

\subsection{Sprint 2: Advanced Features and Hybrid Execution}

Sprint 2 expanded the system's capabilities with advanced language features, a secure database layer, and the hybrid CPU--QPU execution model.

\subsubsection{Advanced Language Features}
\begin{itemize}
    \item \textbf{Advanced Control Flow:} Support for \texttt{while} loops, \texttt{break}, and \texttt{continue} statements.
    \item \textbf{Complex Types:} Support for complex number types and operations.
    \item \textbf{Bitwise Operations:} Implementation of bitwise operators (\&, |, \textasciicircum, \textasciitilde, <<, >>).
    \item \textbf{Subroutines:} Support for defining and calling functions with return values.
\end{itemize}

\subsubsection{PennyLane Integration Expansion}
\begin{itemize}
    \item Controlled gates: CY, CH, CRX, CRY, CRZ, CP, CSWAP, CCZ.
    \item GlobalPhase gate support.
    \item Advanced modifiers: \texttt{negctrl}, \texttt{ctrl(n)}, \texttt{pow(k)}.
\end{itemize}

\subsubsection{User Management and Security (CIA Compliance)}
\begin{itemize}
    \item \textbf{Confidentiality:} Sensitive API keys encrypted at rest using AES-256.
    \item \textbf{Integrity:} User passwords hashed using Bcrypt with per-user salts.
    \item \textbf{Availability:} Role-Based Access Control (RBAC) for User and Admin roles.
\end{itemize}

\subsubsection{Persistence and Database Layer}
\begin{itemize}
    \item \textbf{ORM Models:} Comprehensive models for Users, Projects, Files, Jobs, Simulations, and Achievements.
    \item \textbf{Migrations:} Alembic integration for schema evolution.
    \item \textbf{Data Integrity:} Foreign key constraints and cascade delete policies.
\end{itemize}

\subsection{Sprint 3: Educational Intelligence and Community Features}

Sprint 3 transformed QCanvas from a conversion-and-simulation tool into an intelligent, community-oriented learning environment.

\subsubsection{Interactive Circuit Visualization}
The visualization pipeline in the editor reacts to source changes through a debounced asynchronous parser (500ms debounce) which keeps the interface responsive. The implementation combines the Monaco editor, \texttt{parseCircuitWithCountAsync}, and the D3-based \texttt{CircuitVisualization.tsx} renderer so the circuit diagram updates continuously without blocking typing. A separate 3D path in \texttt{CircuitVisualization3D.tsx} supports richer spatial rendering using \texttt{@react-three/fiber}.

\subsubsection{Quantum Explain It Mode}
The hover-based explanation layer uses \texttt{getHoverForSymbol} in \texttt{lib/quantumHoverSymbols.ts} to feed Monaco hover providers. When a user hovers over a symbol such as \texttt{H}, \texttt{CNOT}, or \texttt{VQE}, the system provides a markdown-rendered popover with the gate matrix, Bloch-sphere implications, and a brief textbook explanation---reducing context switching for learners.

\subsubsection{Gamified Learning and Community Features}
The gamification subsystem in \texttt{gamification\_service.py} implements an event-driven \texttt{Activity Log $\rightarrow$ Achievement Check} architecture. The system features:
\begin{itemize}
    \item \textbf{44 unique achievements} across 6 logic constraint types (\texttt{activity\_count}, \texttt{level\_reached}, \texttt{streak\_days}, \texttt{distinct\_activity\_count}, and hidden Easter Eggs).
    \item Achievement categories: Getting Started \& Social, Mastery, Algorithms, and Hidden.
    \item XP progression system with dynamic level thresholds scaling to Level 50+.
\end{itemize}

The community recipe-sharing workflow is facilitated by \texttt{ShareModal.tsx} and the shared API layer. Users publish circuit snippets annotated with titles, difficulty levels (Beginner/Intermediate/Advanced), framework, and tags with smart auto-completion.

\subsubsection{Extensibility and New Framework Onboarding}
Sprint 3 reinforced the plug-in style design by keeping language support data-driven. New syntax rules, reserved keywords, and framework dictionaries can be added without rewriting the editor pipeline. Monarch tokenizer injection allows bespoke language configurations for \texttt{qasm} (OpenQASM) and \texttt{pythonq} (Quantum Python wrappers).

\subsection{Sprint 4: QSim SDK Integration, Benchmarking, and PyPI Deployment}

Sprint 4 marks the final delivery of the QCanvas FYP project. The sprint focused on production readiness, QSim SDK finalization, comprehensive cross-framework benchmarking, and the public deployment of both SDKs to PyPI.

\subsubsection{QSim SDK and QCanvas SDK -- PyPI Deployment}
Both the QCanvas converter SDK and the QSim simulation SDK have been finalized, packaged, and deployed to the Python Package Index (PyPI). This represents a major milestone: the work produced in this FYP is now accessible to the global quantum computing community.

\begin{itemize}
    \item \texttt{pip install qsim-sdk}: Installs the standalone quantum simulation engine with statevector, density matrix, and stabilizer backends.
    \item \texttt{pip install qcanvas-sdk}: Installs the quantum circuit conversion library, enabling programmatic Qiskit/Cirq/PennyLane to OpenQASM 3.0 conversion.
\end{itemize}

The integration between QCanvas and QSim is seamless: the QCanvas Web IDE uses QSim as its execution backend, while QSim is independently usable as a Python library in scripts, Jupyter notebooks, and research workflows.

\subsubsection{Enhanced Example Library}
The examples page was expanded to include over 40 categorized quantum algorithm implementations across multiple frameworks and categories:
\begin{itemize}
    \item \textbf{Foundational}: Bell State, GHZ State, Quantum Teleportation.
    \item \textbf{Oracle-Based}: Deutsch--Jozsa, Bernstein--Vazirani.
    \item \textbf{Search}: Grover's Algorithm.
    \item \textbf{Variational}: VQE, QAOA.
    \item \textbf{Quantum ML}: XOR Classifier, QRNG.
    \item \textbf{Error Correction}: Bit-Flip Code, Phase-Flip Code.
\end{itemize}

\subsubsection{Production Deployment and Infrastructure}
The QCanvas platform is deployed using Docker Compose with Nginx as the reverse proxy, FastAPI backend, Next.js frontend, PostgreSQL, and Redis containers. The deployment configuration supports horizontal scaling and is AWS-compatible.

\section{Testing and Validation}

\subsection{Testing Strategy}
The verification approach combines three layers: (1) backend regression tests covering the converter, simulator, database layer, and authentication flows; (2) targeted feature checks for Sprint 3 and 4 UI additions; and (3) manual exploratory testing for behavior best evaluated visually, especially circuit diagram rendering and gamification progression.

\subsection{Unit Testing}
Unit testing was employed throughout all sprints to validate individual components in isolation. The testing suite covers:
\begin{itemize}
    \item Conversion correctness for all supported gate types across Cirq, Qiskit, and PennyLane.
    \item OpenQASM 3.0 output validity against the supported subset.
    \item QSim simulation correctness against known quantum state distributions.
    \item Authentication and RBAC logic.
    \item Gamification achievement criteria evaluation.
\end{itemize}

\subsection{Test Coverage}
\begin{itemize}
    \item \textbf{Total Tests:} 304 passing tests with 100\% pass rate.
    \item \textbf{Sprint 1 Tests:} 73 passed tests (110 collected, 33 skipped) covering basic features.
    \item \textbf{Sprint 2 Tests:} 151 tests covering advanced features, database models, and authentication.
    \item \textbf{Integration Tests:} 80 tests covering end-to-end framework conversion and hybrid execution flows.
    \item \textbf{Scope Coverage:} Comprehensive coverage across Cirq, Qiskit, PennyLane, and OpenQASM 3.0 semantics.
\end{itemize}

\section{Evaluation and Benchmarking}

Sprint 4 delivered a comprehensive, reproducible benchmarking study titled \textit{``Cross-Framework Quantum Algorithm Benchmarking via Unified Compilation to OpenQASM 3.0''}, using QCanvas as the unified compilation platform and QSim as the simulation engine. Every algorithm was implemented idiomatically in all three frameworks, compiled to OpenQASM 3.0, and simulated on QSim for a controlled, apples-to-apples comparison.

\subsection{Benchmark Setup}
\begin{itemize}
    \item \textbf{Algorithm Suite:} 12 standard quantum algorithms (Bell State, GHZ State, Quantum Teleportation, Deutsch--Jozsa, Bernstein--Vazirani, Grover's Algorithm, QRNG, VQE, QAOA, QML XOR Classifier, Bit-Flip Code, Phase-Flip Code).
    \item \textbf{Frameworks:} Qiskit, Cirq, PennyLane -- each with idiomatic, native-style implementations.
    \item \textbf{Qubit Range:} 2--8 qubits per algorithm, totaling 136 distinct benchmark configurations.
    \item \textbf{Shots:} 4,096 shots per simulation run; 10 runs per configuration for statistical robustness.
\end{itemize}

\subsection{Performance Evaluation}
\begin{itemize}
    \item \textbf{Conversion Latency:} The average conversion time for a 50-gate Qiskit circuit to OpenQASM 3.0 was measured at \textbf{145ms}, well within the 1500ms target (NFR-P1).
    \item \textbf{Simulation Throughput:} The QSim stabilizer backend simulated 1000 shots of a 20-qubit Clifford circuit in under \textbf{1.2 seconds} (NFR-P2).
    \item \textbf{Visualization Render Time:} D3.js circuit diagram updates averaged \textbf{sub-200ms} for circuits up to 50 gates (NFR-P3).
    \item \textbf{API Scalability:} Load testing on the \texttt{/compile} endpoint showed stable performance up to 50 concurrent requests per second.
    \item \textbf{Round-Trip Accuracy:} Converting standard gates to OpenQASM 3.0 and back preserved all gate parameters with $10^{-10}$ numerical precision.
\end{itemize}
