Chapter 1
Introduction
The product whose requirements are specified in this document is "QCanvas," a unified
quantum simulator platform. This document is intended for project supervisors, devel-
opers, and testers who require a technical overview of the project’s architecture, require-
ments, and design.
1.1 Problem Statement
Despite rapid progress in quantum computing, the developer and learner experience re-
mains fragmented across multiple frameworks, toolchains, and mental models. The ab-
sence of a common, practical interchange format in day-to-day workflows forces re-
searchers and students to learn duplicated concepts with different APIs, impeding col-
laboration and reproducibility. Educational environments further struggle to present im-
mediate feedback and visual intuition while remaining faithful to modern quantum pro-
gramming practices.
QCanvas addresses this gap with a single, standards-aligned pipeline centered on Open-
QASM 3.0. It enables authors to write in Cirq, Qiskit, or PennyLane, convert to Open-
QASM 3.0 as a universal IR, validate the output, and simulate on multiple backends with
live visual feedback in a web IDE. The platform targets education and research, prioritiz-
ing clarity, correctness, and fast iteration.
1.2 Scope
QCanvas shall:
• Provide robust conversion from Cirq, Qiskit, and PennyLane to OpenQASM 3.0.
1
1. Introduction
• Validate generated OpenQASM 3.0 against an explicitly supported subset aligned
with project scope.
• Execute circuits via software simulation backends (statevector, density matrix, sta-
bilizer) with configurable shots and summaries.
• Offer a web-based IDE for editing, conversion, execution, visualization, and sharing
of example circuits.
• Stream progress and results via WebSocket for long-running operations.
Out of scope: pulse-level control, hardware-specific extensions and extern functions, tim-
ing features (delay, duration, box), memory management, and quantum error correction.
1.3 Scope
This section outlines the main modules and their key features within scope.
1.3.1 Conversion Module
• Framework parsers for Cirq, Qiskit, and PennyLane with consistent internal mod-
els.
• OpenQASM 3.0 generation covering types, gates, classical flow, measurement, and
reset (per scope).
• Gate mapping with parameter and broadcast handling; safe fallbacks for unsup-
ported ops.
• Conversion statistics: qubits, depth, gate counts, measurement schema.
• Dual-path strategy: AST-first parsing with safe runtime fallback where AST is in-
sufficient.
• Extensibility hooks to register additional frameworks through a converter factory.
1.3.2 Simulation Module
• Statevector, density matrix, and stabilizer backends.
• Configurable shots; histogram-ready results; optional seed control.
• Lightweight caching for identical executions; pluggable backend interface.
2
1.3 Scope
• Backend capability discovery (e.g., max qubits, supported gates); graceful degrada-
tion on limits.
• Results envelope with counts, metadata (execution time, backend), and optional
state snapshots when available.
1.3.3 Web IDE Module
• Code editor with diagnostics; circuit visualization; results and logs panels.
• Examples browser; project workspace; export/import of QASM.
• Accessibility-minded UI with keyboard-first workflows.
• Real-time progress indicator for conversion and simulation tasks.
• Persistent settings (theme, keybindings) and per-project configuration.
1.3.4 API Module
• REST endpoints for conversion/validation; WebSocket for progress and streaming.
• Structured responses with metadata for reproducibility.
• Endpoint categories: health, convert (single/batch), validate, simulate (single/batch),
backends/frameworks info.
• Error model with machine-readable codes and human-readable messages.
1.3.5 Validation Module
• Syntax and semantic checks; explicit subset conformance for OpenQASM 3.0.
• Helpful error messages with suggested fixes and line/column mapping.
• Static checks for excluded features (timing, pulse-level, extern functions) with clear
guidance.
1.3.6 Assumptions and Dependencies
• Users provide syntactically valid source code for supported frameworks where fea-
sible; invalid code yields diagnostics.

• Browser with modern JavaScript, WebSocket support; stable network connectivity
for API operations.
• Server execution environment with Python 3.11+, FastAPI, and required quantum
libraries.
1.3.7 Constraints
• Maximum circuit size is bounded by backend capabilities and resource limits (mem-
ory/timeouts).
• OpenQASM 3.0 features are limited to the explicitly supported subset (per project
scope).
• No hardware integration; simulation-only execution path.
1.4 User Classes and Characteristics
User Class Characteristics
Beginners Prefer guided learning, clear feedback, visualizations, and tem-
plates to start quickly.
Intermediate Learners Seek examples that transition from basic to advanced topics
with immediate validation.
Researchers Need accuracy, batch runs, parameter sweeps, and cross-
framework comparison with consistent outputs.
Educators Require curated examples, exercises, progress indicators, and
shareable workspaces.
Developers/Testers Need diagnostics, logs, reproducibility, and deterministic
seeds for CI workflows.

Chapter 2
Project Requirements
This chapter describes the functional and non-functional requirements, followed by use-
cases and diagrams, matching the target pattern. Requirements align strictly with the
defined project scope and express system behavior rather than development iterations or
phases.
2.1 Functional Requirements
2.1.1 Conversion Module
• The system must parse Cirq, Qiskit, and PennyLane circuits.
• The system must generate OpenQASM 3.0 for supported constructs per project
scope.
• The system must handle measurement and reset instructions.
• The system must expose conversion statistics (qubits, depth, gate counts).
• The system must validate generated OpenQASM 3.0 against the supported subset
and provide diagnostics.
2.1.2 Simulation Module
• The system must support statevector, density matrix, and stabilizer backends.
• The system must allow configurable shots and return counts and metadata.
• The system must cache identical executions.
