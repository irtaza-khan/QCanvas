\chapter{System Overview}

The system architecture of QCanvas follows a modular layered design separating conversion, validation, simulation, and UI concerns. This design ensures scalability, maintainability, and flexibility while supporting the hybrid CPU–QPU execution model.

\section{Architectural Design}

QCanvas employs a hybrid architecture where the Next.js frontend handles UI and simple operations, while the FastAPI backend manages complex quantum computations. QCanvas orchestrates compilation and validation; QSim executes circuits on simulation backends.

\subsection{Layered Responsibilities}
\begin{itemize}
    \item \textbf{Presentation Layer (Next.js)}: Editor UI, circuit visualization, results dashboards, accessibility features, and client-side state management.
    \item \textbf{API Layer (FastAPI)}: REST endpoints for conversion/validation/simulation, WebSocket for progress streams, structured error handling, and request validation.
    \item \textbf{Conversion Layer (QCanvas)}: AST parsing, intelligent gate mapping, OpenQASM 3.0 generation, validation, analytics, and logging.
    \item \textbf{Execution Layer (QSim)}: Simulation backends (statevector, density matrix, stabilizer), job scheduling, result aggregation, caching, and metrics.
    \item \textbf{Persistence Layer}: PostgreSQL for user data and projects (future), Redis for caching and sessions (future).
    \item \textbf{Observability Layer}: Health checks, Prometheus metrics, execution time histograms, and structured logging.
\end{itemize}

\subsection{Technology Stack}

\subsubsection{Frontend Technologies}
\begin{itemize}
    \item Next.js 14 with App Router for server-side rendering and routing
    \item React 18 with TypeScript for type-safe UI development
    \item Tailwind CSS for styling with custom quantum-themed design
    \item Monaco Editor for code editing with syntax highlighting
    \item Zustand for lightweight state management
    \item Chart.js for measurement histogram visualization
    \item WebSocket client for real-time updates
\end{itemize}

\subsubsection{Backend Technologies}
\begin{itemize}
    \item Python 3.11+ for backend logic
    \item FastAPI for high-performance async API framework
    \item Uvicorn ASGI server for production deployment
    \item Pydantic for data validation and serialization
    \item SQLAlchemy ORM with PostgreSQL (future)
    \item Redis for caching and session management (future)
\end{itemize}

\subsubsection{Quantum Computing Stack}
\begin{itemize}
    \item Cirq 1.0+ for Google quantum framework support
    \item Qiskit 0.44+ for IBM quantum framework support
    \item PennyLane 0.33+ for Xanadu quantum framework support
    \item Custom OpenQASM 3.0 generator (QASM3Builder)
    \item Custom simulation backends (statevector, density matrix, stabilizer)
\end{itemize}

\subsection{Hybrid CPU–QPU Execution Model}

\subsubsection{Orchestration Boundary}
\begin{itemize}
    \item \textbf{CPU Responsibilities}: Circuit construction, parameter preparation, AST parsing, OpenQASM generation, validation, pre/post-processing, result aggregation.
    \item \textbf{QPU/Simulator Responsibilities}: Gate-level circuit execution, statevector/density matrix evolution, measurement sampling, state snapshot generation.
\end{itemize}

\subsubsection{Data Flow}
\begin{enumerate}
    \item User writes circuit in source framework (Qiskit/Cirq/PennyLane)
    \item Frontend sends code to backend via REST API
    \item Backend parses code using AST or runtime execution
    \item QCanvas generates OpenQASM 3.0 intermediate representation
    \item Validation checks conformance to supported subset
    \item QSim executes circuit on selected backend
    \item Results aggregated and returned via API
    \item Frontend visualizes results (histograms, statistics)
    \item WebSocket provides real-time progress updates
\end{enumerate}

\subsubsection{Execution Scope}
\begin{itemize}
    \item Simulation-first: All execution through software simulators
    \item No pulse-level control or timing features
    \item Future pluggable QPU endpoints behind same API contract
    \item Intelligent caching for identical circuit executions
\end{itemize}

\section{Data Design}

\subsection{Information Domain Transformation}

The conceptual entities from the domain model are transformed into runtime data structures and serialized formats:

\subsubsection{Core Entities}
\begin{itemize}
    \item \textbf{Circuit}: Framework source code, parsed AST, OpenQASM 3.0 representation, metadata, and statistics (qubits, depth, gate counts).
    \item \textbf{ConversionRequest}: Source framework, target framework or QASM, source code, optimization options, validation flags.
    \item \textbf{ConversionResult}: Generated OpenQASM 3.0 code, conversion statistics, warnings, errors, execution time.
    \item \textbf{SimulationJob}: OpenQASM code, selected backend, shots, seed, execution status, progress percentage.
    \item \textbf{SimulationResult}: Measurement counts, statevector/density matrix (optional), execution time, metadata.
    \item \textbf{Project}: Files collection, active file reference, project settings, theme preferences.
\end{itemize}

\subsection{Data Organization and Processing}

\begin{longtable}{p{0.22\textwidth} p{0.7\textwidth}}
\toprule
\textbf{Layer} & \textbf{Data Handled / Description} \\
\midrule
Input Layer & Source code (Python/QASM), examples, editor configuration (JSON), user preferences \\
Parsing Layer & Python AST, circuit operations, gate definitions, qubit mappings \\
Conversion Layer & CircuitAST intermediate format, gate mappings, OpenQASM 3.0 text, diagnostics with line/column info \\
Validation Layer & Syntax errors, semantic errors, unsupported feature warnings, subset conformance checks \\
Simulation Layer & Backend selection (statevector/density/stabilizer), shots configuration, execution parameters, caching keys \\
Results Layer & Measurement counts, histograms, statevector arrays, execution metadata \\
Rendering Layer & Circuit diagrams (SVG), charts (Chart.js), formatted statistics (client-rendered) \\
Persistence Layer & Project files, conversion history, results cache, user sessions (file- or memory-backed, DB in future) \\
\bottomrule
\end{longtable}

\subsection{Data Storage Mechanisms}

\subsubsection{Current Implementation}
\begin{itemize}
    \item In-memory storage for active sessions and temporary results
    \item File-based storage for example circuits and templates
    \item Frontend local storage for user preferences and theme
\end{itemize}

\subsubsection{Future Implementation}
\begin{itemize}
    \item PostgreSQL for user accounts, projects, and circuit library
    \item Redis for session management and result caching
    \item SQLAlchemy ORM for database abstraction
    \item Migration system for schema evolution
\end{itemize}

\subsection{API Data Models}

All API endpoints use Pydantic models for validation:

\subsubsection{Request Models}
\begin{itemize}
    \item ConversionRequest: source\_framework, target\_framework, source\_code, style, validate
    \item SimulationRequest: qasm\_code, backend, shots, seed, noise\_model
    \item FileRequest: name, content, language, project\_id
\end{itemize}

\subsubsection{Response Models}
\begin{itemize}
    \item ConversionResponse: success, qasm\_code, stats, warnings, errors, timestamp
    \item SimulationResponse: success, results, execution\_time, backend\_info, timestamp
    \item ErrorResponse: error\_code, message, details, suggestions
\end{itemize}
