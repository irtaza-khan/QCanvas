\chapter{Introduction}

The field of quantum computing is rapidly evolving, shifting from theoretical research to practical application. As hardware capabilities grow, so does the need for accessible, standardized software tools that can bridge the gap between abstract quantum concepts and executable code. "QCanvas" is proposed as a unified quantum simulation platform designed to address this need by providing a standards-aligned environment centered on OpenQASM 3.0.

This document details the software requirements for QCanvas, intended for developers, researchers, and supervisors. It outlines the architectural design, system capabilities, and constraints that define the project.

\section{Problem Statement}

The current quantum development ecosystem is fragmented. Learners and researchers are forced to navigate a disconnect between multiple software frameworks (such as Qiskit, Cirq, and PennyLane) and the underlying hardware behavior. Key challenges include:

\begin{itemize}
    \item \textbf{Fragmentation:} Developers must learn duplicate concepts across different APIs, making it difficult to port or compare experiments.
    \item \textbf{Lack of Standardization:} The absence of a transparent intermediate representation in many workflows obscures how high-level code maps to actual quantum operations.
    \item \textbf{Steep Learning Curve:} Educational tools often lack immediate visual feedback, while professional tools are too complex for beginners, widening the gap between theory and practice.
\end{itemize}

QCanvas aims to solve these problems by introducing a unified interface where users can write in their preferred framework, but understand and execute their logic through a common, standardized lens (OpenQASM 3.0).

\section{Project Objectives}

The primary goal of QCanvas is to democratize access to quantum programming through a simulation-first approach. The specific objectives are:

\begin{itemize}
    \item \textbf{To Unify Quantum Workflows:} Create a single platform capable of parsing and converting code from major frameworks (Cirq, Qiskit, PennyLane) into a universal format (OpenQASM 3.0).
    \item \textbf{To Enhance Educational Clarity:} Provide a web-based IDE that offers immediate feedback, converting abstract Python code into clear quantum assembly and visual circuit diagrams.
    \item \textbf{To Ensure Standard Compliance:} Validate all generated code against a strict OpenQASM 3.0 subset to ensure correctness and portability.
    \item \textbf{To Enable Hybrid Simulation:} Implement a robust architecture where classical orchestration (CPU) and quantum execution (Simulators) are clearly separated, mimicking real-world hybrid quantum-classical systems.
\end{itemize}

\section{Project Scope}

The scope of QCanvas is defined to deliver a functional, robust simulation environment within the project timeline.

\subsection{In Scope}

The system shall:
\begin{itemize}
    \item \textbf{Conversion Engine:} Robustly parse Python-based quantum circuits and convert them to OpenQASM 3.0 AST and source code.
    \item \textbf{Validation Layer:} Semantic analysis to ensure generated code adheres to the supported gate set and control flow limitations.
    \item \textbf{Simulation Backends:} Support for Statevector, Density Matrix, and Stabilizer simulations with configurable shot counts.
    \item \textbf{Interactive IDE:} A Monaco-based web editor for writing code, visualizing results (histograms), and managing projects.
    \item \textbf{Real-Time Feedback:} WebSocket-based streaming for long-running batch conversions or heavy simulations.
\end{itemize}

\subsection{Out of Scope}

To ensure feasibility, the following are explicitly excluded:
\begin{itemize}
    \item \textbf{Hardware Execution:} Direct integration with physical QPUs (e.g., IBM Quantum Experience) is not part of this release.
    \item \textbf{Pulse-Level Control:} Low-level control of microwave pulses (OpenPulse) is outside the scope.
    \item \textbf{QEC Protocols:} Specialized quantum error correction encoding/decoding layers are not implemented.
    \item \textbf{Physical Qubit Mapping:} Manual addressing of physical qubit layouts (e.g., \texttt{q[0] -> \$0}) is not supported.
\end{itemize}

\section{Overview of System Modules}

The system is architected into modular components to ensure scalability and maintainability:

\begin{itemize}
    \item \textbf{Conversion Module:} The core engine responsible for AST parsing, framework interoperability, and code generation.
    \item \textbf{Simulation Module:} The execution engine handling the mathematics of state evolution and measurement.
    \item \textbf{Validation Module:} A gatekeeper ensuring code correctness before execution.
    \item \textbf{Web IDE Module:} The frontend interface handling user interaction, visualization, and state management.
    \item \textbf{API Module:} The communication layer exposing REST and WebSocket endpoints.
\end{itemize}

\section{Assumptions, Dependencies, and Constraints}

\subsection{Assumptions}
It is assumed that users have basic familiarity with Python syntax. The system assumes a stable modern browser environment (Chrome/Firefox/Edge) for the IDE.

\subsection{Dependencies}
Key external dependencies include the \texttt{Python 3.11+} runtime, \texttt{FastAPI} for the backend, \texttt{Next.js} for the frontend, and the quantum libraries \texttt{Cirq}, \texttt{Qiskit}, and \texttt{PennyLane}.

\subsection{Constraints}
System performance is constrained by host memory for Statevector simulations (scaling exponentially with qubit count). API response times are targeted at under 2 seconds for standard conversion tasks.
